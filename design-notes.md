# 自定义 Skill 设计思考笔记

> 记录于 2026-06-21，整理自一次关于"如何做一个自己能用、别人也能用的自定义 Skill"的讨论。

## 一、Skill 是什么

Skill（技能）是 Claude Code 里一种「可被按需调用的专门能力包」——预先写好的、带有专门指令和领域知识的工作流模块。当任务匹配某个 Skill 时即被调用，按其中的步骤和规则执行。

- 终端里输入 `/<名字>` 调用的，本质就是 Skill；「slash command」和「Skill」基本是同一回事。
- 调用方式两种：① 用户主动 `/技能名`；② Claude 根据 `description` 自动匹配触发。

## 二、Skill 的文件结构

一个 Skill = 一个文件夹 + 一个 `SKILL.md`：

```
~/.claude/skills/<skill名>/SKILL.md      ← 全局可用
<项目>/.claude/skills/<skill名>/SKILL.md  ← 仅该项目可用
```

`SKILL.md` 核心 = frontmatter + 正文：

```markdown
---
name: my-skill
description: 一句话说明"什么场景下该触发我"——自动匹配的依据
---

正文：被调用后要按什么步骤、什么规则做事。
可引用其他文件、规定输出格式、限定流程。
```

关键点：
- **`description` 是触发的灵魂**：写得越精准，越能在对的时候被用上。
- **正文 = 操作手册**：步骤清单、检查项、模板、禁止事项。
- **可拆分文件**：复杂 Skill 可在文件夹里放额外 `.md` / 脚本 / 模板，正文引用、按需加载。
- **可限定工具**：能声明这个 Skill 只允许用哪些工具。

## 三、关键认知：私人 Skill vs 可分发 Skill

最初的着眼点太小——只盯着绑死在个人场景上的工具（个人目录、个人项目、个人约定），换个人就废。真正要做的是把**一种能力 / 方法论抽象出来**，做成谁拿去都能用的工具。

| | 私人 Skill | 可分发 Skill（目标） |
|---|---|---|
| 绑定的是 | 个人路径、项目、约定 | 一套**方法论 / 流程**本身 |
| 换个人 | 直接废掉 | 照样能用 |
| 价值来源 | 省重复交代的麻烦 | 把"高手怎么做某事"固化给所有人 |
| 输入假设 | "在我的环境里" | "给我一个任意的 X，我来处理" |

## 四、做通用 Skill 的设计原则

1. **绑流程，不绑环境**：不假设任何人的目录结构、命名、已装工具；一切靠运行时探测。
2. **输入要泛化**：从"处理*我的* X"变成"给我*任意一个* X，我来处理"。
3. **方法论才是核心资产**：真正值钱的是把"某领域高手的思考步骤"写进正文。

## 五、判断一件事值不值得做成 Skill

满足两三条即值得：
1. 重复发生（同类任务一周做好几次）
2. 有固定套路（步骤/规则差不多）
3. 每次都要重新交代背景
4. 做错有代价（漏步骤就返工）

## 六、候选方向（绑方法论、换谁都能用）

- **`source-spelunk`（陌生源码考古）**：丢给它任意陌生开源项目（C/C++/ObjC/Rust），用"定位入口 → 画依赖 → LLDB 下断点实测 → 绝不凭记忆下结论"的方法论帮人快速吃透。
- **`explain-like-source`（源码级技术讲解）**：给一个技术概念，产出"贴真源码 + 实测验证 + 交互图"的高质量讲解。
- **`debug-from-binary`（从二进制/崩溃反推）**：给 crash log 或可疑行为，用系统化调试/逆向流程定位根因。

## 七、已定案：做 `clean-commit`（2026-06-22）

第六节三个候选方向均搁置，最终方向来自更早一次未保存的会话——**做一个"提交 GitHub"的 skill，规范提交流程**。锚定结论：

- **目标人群**：更泛的程序员（语言无关），不绑任何人的目录/工具链，一切靠运行时探测。
- **核心方法论**：8 步流程——探测环境 → 读懂 diff → 扫密钥 → 清临时文件 → 逻辑切分 → 跟随仓库已有 message 风格 → 提交前验证 → 仅按需 push/PR。
- **三条拍板**：① message 风格运行时探测**跟随**仓库、不强加；② **不**默认加 `Co-Authored-By`，可配置；③ 名字 `clean-commit`。
- **约定式提交附录**：基于 Conventional Commits v1.0.0 / Angular 规范 / SemVer 2.0.0 / RFC 2119，仅在仓库已用或强制（检测 `commitlint.config.*`/`.czrc`/`commitizen`/`commit-msg` hook）或用户要求时按附录精确执行——解决"会跟但跟不准"的缺口。
- **落地位置**：`~/.claude/skills/clean-commit/SKILL.md`（全局）。
- **验收**：临时仓库冒烟测试通过（密钥拦截 / 强制约定检测 / `type(scope):` 风格跟随均 OK）。
